##### 13/06/2026

# Introduzione del sistema di effort detection basato su Ruptures

## Nuovo motore di rilevamento effort tramite changepoint detection con selezione dinamica del metodo di analisi.

### ✨ Nuove funzionalità (Added)

- Integrata la libreria `ruptures` per il rilevamento automatico degli effort tramite algoritmi di changepoint detection.
- Aggiunto il nuovo modulo `ruptures_analyzer.py` contenente:
  - `RupturesConfig`
  - `detect_efforts_ruptures()`
  - `ruptures_config_from_dict()`
- Implementato un nuovo workflow di analisi selezionabile tramite parametro `analysis_method` con supporto a:
  - `ruptures`
  - `legacy`
- Aggiunta una nuova configurazione avanzata per il metodo Ruptures:
  - modello (`l2`, `l1`, `rbf`)
  - penalty
  - durata minima segmento
  - smoothing del segnale
  - soglia minima `%CP`
  - merge gap temporale
  - merge per differenza percentuale di potenza
- Introdotto un algoritmo di merge intelligente degli effort che combina segmenti adiacenti solo se:
  - la distanza temporale è inferiore alla soglia configurata;
  - la differenza di potenza media rientra nella tolleranza definita.
- Aggiunto supporto ai filtri Jinja2 personalizzati tramite il nuovo modulo `jinja_filters.py`.
- Registrato il filtro `duration` per la formattazione automatica delle durate in formato `mm:ss` o `h:mm:ss`.
- Esposizione dell'istanza `Jinja2Templates` tramite `app.state.templates` per utilizzo condiviso all'interno dell'applicazione.

### 🔄 Modifiche (Changed)

- Rifattorizzato il flusso di upload FIT per supportare due pipeline di rilevamento effort indipendenti:
  - pipeline legacy basata su finestre mobili;
  - pipeline Ruptures basata sulla segmentazione del segnale.
- Aggiornata la struttura dati di sessione per memorizzare:
  - metodo di rilevamento utilizzato;
  - configurazione Ruptures;
  - configurazione effort legacy compatibile con il codice esistente.
- Semplificato `analysis_config.py` eliminando la configurazione globale `AnalysisConfig` e le strutture non più utilizzate.
- Ridotte le dataclass di configurazione alle sole impostazioni effettivamente necessarie al sistema di detection.
- Aggiunto il re-export di `RupturesConfig` per mantenere compatibilità con gli import esistenti.
- Uniformato il logging utilizzando placeholder parametrizzati invece di string interpolation diretta.

### 🧠 Logica e stato

- Il motore Ruptures esegue una pipeline composta da:
  1. smoothing opzionale del segnale di potenza;
  2. downsampling automatico a 1 Hz;
  3. segmentazione tramite algoritmo PELT;
  4. classificazione dei segmenti rispetto alla soglia CP;
  5. merge controllato dei segmenti compatibili.
- Gli effort rilevati vengono salvati insieme alla configurazione completa che ha generato il risultato.
- Introdotta una configurazione compatibile (`EffortConfig`) per garantire il funzionamento dei moduli esistenti che utilizzano ancora il modello legacy.

### 🐞 Correzioni (Fixed)

- Migliorata la gestione degli errori durante:
  - upload e salvataggio dei file FIT;

- Rafforzata la validazione dei parametri inviati dal client con controlli specifici per ciascun metodo di analisi.
- Migliorata la pulizia dei file temporanei tramite utilizzo sistematico di `unlink(missing_ok=True)`.
- Ottimizzata la gestione degli upload limitando casi di file vuoti, file eccessivamente grandi e parametri potenzialmente abusivi.
- Corretto il comportamento del rate limiting con validazione più robusta dei parametri provenienti da variabili d'ambiente.

### 🎨 Miglioramenti UI

- Ridisegnata la schermata di upload introducendo un selettore visuale del metodo di detection.
- Aggiunti pannelli configurabili separati per:
  - **Ruptures Detection**
  - **Legacy Detection**
- Introdotto uno slider dedicato alla regolazione della penalty del modello Ruptures con aggiornamento immediato del valore.
- Aggiunte descrizioni contestuali e suggerimenti operativi per i parametri di segmentazione.
- Implementata la persistenza delle impostazioni utente tramite `localStorage`.
- Aggiornato il sistema di drag & drop mantenendo automaticamente le configurazioni precedentemente utilizzate.
- Migliorata l'organizzazione dei parametri avanzati con sezioni dedicate e visibilità dinamica in base al metodo selezionato.

### ⚙️ Infrastruttura

- Registrazione centralizzata dei filtri Jinja all'avvio dell'applicazione.
- Predisposta l'architettura per l'introduzione futura di ulteriori algoritmi di effort detection senza impatti sui moduli consumer.

### Risultato finale

L'applicazione supporta ora due modalità indipendenti di rilevamento degli effort. Gli utenti possono scegliere tra il metodo storico basato su finestre mobili e un nuovo sistema avanzato basato su changepoint detection, configurando direttamente dall'interfaccia i parametri di segmentazione e ottenendo un controllo più preciso sul processo di identificazione degli effort.
